## Warnings

### WR-01: Parser de `.env.local` não remove aspas de valores

**File:** `tests/e2e/helpers/env.ts:18-27`  
**Issue:** O parser lê o valor como tudo após o `=` e remove apenas espaços em branco. Se o usuário escrever `NEXT_PUBLIC_SUPABASE_URL="http://127.0.0.1:54321"` no `.env.local` seguindo a convenção comum de colocar aspas em variáveis com caracteres especiais, o valor carregado será `"http://127.0.0.1:54321"` — com aspas literais. Isso causa falha silenciosa ao instanciar o cliente Supabase (URL malformada) ou ao preencher credenciais Google (email com aspas é inválido). O erro resultante (`invalid URL`) não aponta para o parser, dificultando o diagnóstico.

**Fix:** Adicionar remoção de aspas simples/duplas envolventes após o `trim()`:

```typescript
const rawValue = trimmedLine.slice(separatorIndex + 1).trim()
const value =
  (rawValue.startsWith('"') && rawValue.endsWith('"')) ||
  (rawValue.startsWith("'") && rawValue.endsWith("'"))
    ? rawValue.slice(1, -1)
    : rawValue
if (key && !process.env[key]) {
  process.env[key] = value
}
```

## Info

### IN-01: `normalizeSameSite` não trata valores booleanos de `sameSite`

**File:** `tests/e2e/auth.setup.ts:154-162`  
**Issue:** A função `normalizeSameSite` aceita `unknown` mas somente verifica strings `'Strict'`, `'strict'`, `'None'` e `'none'` — defaultando para `'Lax'` em todos os outros casos. A opção `sameSite` nos cookies do `@supabase/ssr` pode ser um booleano (`true` ou `false`). Se `true` for passado, o default `'Lax'` pode não corresponder à intenção original do cookie (que normalmente indicaria `'Strict'`).

**Fix:** Adicionar tratamento explícito para booleanos e para a variante case-insensitive `'lax'`:

```typescript
const normalizeSameSite = (raw: unknown): 'Lax' | 'Strict' | 'None' => {
  if (raw === true || raw === 'Strict' || raw === 'strict') return 'Strict'
  if (raw === 'None' || raw === 'none') return 'None'
  return 'Lax'
}
```

### IN-02: `.first()` reduz precisão do localizador de heading em `@auth-02`

**File:** `tests/e2e/auth.spec.ts:15`  
**Issue:** `page.getByRole('heading', { name: 'Meu Perfil' }).first()` silencia o caso em que mais de um heading "Meu Perfil" existe na página — um indicador de regressão de UI que o teste deveria detectar. O `.first()` foi provavelmente adicionado para evitar falha por múltiplos matches, mas encobre o problema em vez de corrigi-lo.

**Fix:** Remover `.first()` e, se houver múltiplos headings, ajustar o escopo com um localizador mais preciso (ex: `page.getByRole('main').getByRole('heading', { name: 'Meu Perfil' })`). Se o heading é único na tela, o selector sem `.first()` é suficiente e falhará corretamente se a contagem mudar:

```typescript
await expect(page.getByRole('heading', { name: 'Meu Perfil' })).toBeVisible()
```

### IN-03: Cópia de UI hardcoded em assertion de `@auth-04c`

**File:** `tests/e2e/auth.spec.ts:39-42`  
**Issue:** O texto `'Você precisará entrar com Google novamente para acessar suas finanças.'` está embutido diretamente na assertion. Qualquer alteração no copywriting do AlertDialog quebra o teste sem que a falha aponte claramente para uma regressão funcional vs. uma mudança de texto intencional. O mesmo vale para `'Sair da conta?'` na linha 37.

**Fix (opção pragmática):** Extrair as strings para constantes no topo do arquivo de spec com um comentário indicando que são cópias de UI que precisam ser mantidas em sincronia:

```typescript
// Cópias de UI — manter em sincronia com o AlertDialog de logout
const LOGOUT_DIALOG_TITLE = 'Sair da conta?'
const LOGOUT_DIALOG_BODY =
  'Você precisará entrar com Google novamente para acessar suas finanças.'
```

Alternativamente, usar `toContainText` com substrings menos frágeis (ex: `'entrar com Google'`) se a frase exata não for parte do contrato de acceptance.

### IN-04: Inconsistência de comentários em `.env.example` para variáveis E2E opcionais

**File:** `.env.example:13,24-25`  
**Issue:** `E2E_AUTH_MODE` está comentado (`# E2E_AUTH_MODE=password`) enquanto `E2E_GOOGLE_TEST_EMAIL` e `E2E_GOOGLE_TEST_PASSWORD` estão descomentados com valores vazios. As três variáveis são igualmente opcionais para E2E. A inconsistência pode levar um desenvolvedor a achar que as duas últimas são obrigatórias, ou a questionar por que o tratamento difere. Funcionalmente não é um bug (o parser em `helpers/env.ts` respeita `!process.env[key]`), mas cria ambiguidade.

**Fix:** Unificar o estilo — comentar as três variáveis opcionais com exemplo de valor para deixar claro que são opcionais e como preenchê-las:

```dotenv
# Opcional — E2E authentication mode. Default: password
# E2E_AUTH_MODE=password

# Opcional — Google credentials for automated OAuth setup (E2E_AUTH_MODE=google)
# E2E_GOOGLE_TEST_EMAIL=sua-conta@gmail.com
# E2E_GOOGLE_TEST_PASSWORD=sua-senha
```
