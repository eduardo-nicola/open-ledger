# Phase 01.1 — Technical research (E2E clarity & confidence)

**Date:** 2026-04-17

## Objective

Responder: *o que precisamos saber para planejar bem a fase 01.1?* — documentação de execução, duas variações Google (A/B), modo password como debug, higiene de tags e mensagens de falha acionáveis, sem expandir cobertura funcional.

## Findings

### 1. Estado atual do runner

- `playwright.config.ts` usa `baseURL: 'http://127.0.0.1:3000'` (não `localhost`); documentação deve repetir o mesmo host para evitar mismatch de cookies.
- Projeto `setup` roda `auth.setup.ts`; `chromium` depende de `setup` e aplica `storageState` em `tests/.auth/user.json`.
- `E2E_AUTH_MODE` padrão em código é `password` (`auth.setup.ts` linha com `?? 'password'`), enquanto o produto e a iniciativa definem **Google OAuth real** como caminho de confiança — a divergência é resolvida por **documentação explícita** e comentários no setup, não obrigatoriamente invertendo o default (que quebraria ambientes sem segredos Google).

### 2. Variação A — Google já autenticado no browser

Opções viáveis (escolha do executor, qualquer uma satisfaz D-03):

| Abordagem | Prós | Contras |
|-----------|------|---------|
| `storageState` gerado uma vez após login manual/headed e reutilizado | Simples, reproduzível localmente | Estado pode expirar; não commitar tokens |
| `launchOptions` com `userDataDir` persistente no setup | Mantém cookies Google entre execuções | Caminho local por máquina; cuidado com paths em CI |
| Perfil Chrome dedicado fora do repo | Bom para dev humano | Documentação precisa de passos claros |

**Validação D-03-A:** após clicar "Entrar com Google", URL deve chegar em `/profile` sem re-digitar email/senha quando o IdP já reconhece a sessão.

### 3. Variação B — Google não logado

- `runGoogleManualSetup` + `tryGoogleCredentialLogin` já cobrem fluxo UI + preenchimento opcional via `E2E_GOOGLE_TEST_EMAIL` / `E2E_GOOGLE_TEST_PASSWORD`.
- Com `E2E_AUTH_MODE=google`, `playwright.config.ts` força `channel: 'chrome'`, `headless: false` e flags anti-automação — necessário para fluxo Google estável em muitos ambientes.

### 4. CI e Google

- Login Google em CI é frequentemente bloqueado ou flaky (anti-bot, CAPTCHA, device trust).
- Alinhado a **D-07**: documentar explícitamente job opcional / execução local obrigatória / segredos no GitHub, em vez de assumir verde em toda PR sem credenciais.

### 5. Tags duplicadas (`auth.spec.ts`)

- Quatro testes compartilham `@auth-04` com asserções diferentes — viola TST-04.
- Correção: sufixos (`@auth-04a` …) **mapeados em tabela** no README para sub-capacidades de AUTH-04 (perfil, read-only, diálogo logout, redirect).

### 6. Scripts npm

- `test:e2e:smoke` → `--grep @smoke`
- `test:e2e:auth` → `--grep @auth`
- `test:e2e:google` → força `E2E_AUTH_MODE=google` + `@smoke`

Semântica: `auth` é superconjunto por grep (pega `@auth-01`, `@auth-04a`, etc.); `smoke` restringe a cenários marcados `@smoke` — documentar isso evita interpretação errada.

## Validation Architecture

> Nyquist / Dimension 8 — como validar esta fase sem ambiguidade.

**Framework:** Playwright 1.59 (`@playwright/test`)

**Arquivos de config:** `playwright.config.ts` (raiz)

**Comandos:**

| Uso | Comando |
|-----|---------|
| Smoke rápido (padrão local/CI sem Google) | `npm run test:e2e:smoke` (com `E2E_AUTH_MODE` não definido ou `password`) |
| Todos cenários com tag auth | `npm run test:e2e:auth` |
| Smoke com setup Google | `npm run test:e2e:google` |
| Suite completa | `npx playwright test` |

**Pré-requisitos de ambiente:** `supabase start` + `supabase db reset` (seed), `.env.local` com URLs Supabase, para Google adicional `E2E_AUTH_MODE=google` + opcionalmente `E2E_GOOGLE_TEST_EMAIL` / `E2E_GOOGLE_TEST_PASSWORD`.

**Amostragem recomendada:**

- Após cada plano que toca `tests/e2e/*.ts` ou `playwright.config.ts`: `npm run test:e2e:smoke` com `E2E_AUTH_MODE=password`.
- Após alterações só em documentação: `grep` / revisão manual do README; smoke opcional.
- Antes de considerar a fase completa: smoke verde local; validação Google (Variação A ou B) **documentada como verificação manual ou job opcional**, não gate obrigatório em CI sem segredos.

**Manual-only:**

- Fluxo Google completo sem credenciais automáticas (login 100% humano).
- Confirmação visual CAPTCHA / bloqueio de conta — fora do escopo automatizado.
