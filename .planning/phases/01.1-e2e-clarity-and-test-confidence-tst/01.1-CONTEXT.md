# Phase 01.1: E2E clarity and test confidence (TST) - Context

**Gathered:** 2026-04-17  
**Status:** Ready for planning

<domain>
## Phase Boundary

Esta fase trata apenas de **clareza, contrato de execução e confiabilidade** dos testes E2E já existentes (Playwright): documentação, matriz de comandos/ambiente, setup de autenticação alinhado ao produto (Google OAuth real), duas variações explícitas de “estado da sessão Google” antes de rodar a suíte autenticada, higiene de tags e mensagens de falha acionáveis. **Não** expande cobertura funcional para Fases 2+.

</domain>

<decisions>
## Implementation Decisions

### Autenticação nos testes (caminho definitivo)

- **D-01:** O caminho **oficial e definitivo** de E2E autenticado é **login Google real** (fluxo OAuth na UI), **não** simular sessão apenas via `signInWithPassword` + injeção de cookies como substituto do comportamento do usuário. O fluxo por API/cookies pode permanecer apenas como **ferramenta de debug opcional** se o plano documentar isso explicitamente — nunca como padrão silencioso.
- **D-02:** Existe **uma conta Google dedicada** aos testes, **sem 2FA**, para reduzir atrito e tempo humano; credenciais via **segredos de ambiente** (ex.: `E2E_GOOGLE_TEST_EMAIL` / `E2E_GOOGLE_TEST_PASSWORD` já usados em `auth.setup.ts`), nunca commitadas.
- **D-03:** Duas **variações de setup** devem ser suportadas e documentadas:
  - **Variação A — Google já logado:** o contexto/navegador já possui sessão Google ativa (perfil persistente, `storageState` pré-gerado com Google, ou equivalente); o setup deve **validar** que após “Entrar com Google” o usuário chega em `/profile` **sem** exigir digitar email/senha Google de novo quando o IdP já reconhece a sessão.
  - **Variação B — Google ainda não logado:** sessão Google **ausente** ou inválida; o setup deve executar o fluxo completo (incluindo preenchimento automatizado quando as env vars estiverem presentes, como em `tryGoogleCredentialLogin`, ou passos manuais quando não estiverem), até `/profile`, e então persistir `storageState` para os testes dependentes.
- **D-04:** **Depois** do setup de autenticação concluído com sucesso, executam-se **os demais testes já existentes** (projetos `chromium` com `storageState`, `@smoke`, `rls`, `auth-unauthenticated`, etc.) **sem** mudar o escopo funcional destes testes nesta fase — apenas garantindo que a **origem da sessão** obedece a D-01–D-03.

### Documentação e contrato (alinha TST-01 … TST-07)

- **D-05:** `tests/e2e/README.md` (ou seção equivalente) deve descrever: pré-requisitos (`supabase`, seed se ainda necessário para dados do perfil), variáveis obrigatórias, **matriz** “Variação A | Variação B × comando npm × CI/local”, e quando usar headed vs headless.
- **D-06:** Tags `@auth-XX` / `@smoke`: **um identificador de requisito por cenário** ou sufixos (`@auth-04a`) com tabela no README mapeando para AUTH-XX do roadmap (fecha TST-04/TST-05).

### CI e custo operacional

- **D-07:** Login Google real em CI pode ser **instável ou bloqueado** por anti-bot; a decisão do produto é priorizar **fidelidade ao login real**. O planejamento deve prever **job/documentação explícitos** (ex.: workflow manual, segredos no GitHub Actions, ou execução local obrigatória antes do merge) — detalhe técnico fica a critério do planner (ver Claude's Discretion).

### Claude's Discretion

- Mecanismo exato da Variação A (perfil persistente Playwright vs `storageState` check-in vs artefato CI).
- Default de `E2E_AUTH_MODE` após a fase (hoje `password`); a intenção é **Google como definitivo** — o planner pode renomear env ou inverter default desde que D-01–D-04 sejam respeitados e documentados.
- Estratégia concreta para PR checks quando Google em CI não for viável (ex.: job opcional, label `e2e-manual`).

</decisions>

<specifics>
## Specific Ideas

- Conta Google de teste **sem 2FA**, só para E2E.
- Quer **confiança** no sentido de que o teste passou por um **login efetivo** (Variação B) ou por uma sessão Google já estabelecida (Variação A), e só então roda o restante da suíte.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets

- `runGoogleManualSetup`, `tryGoogleCredentialLogin`, `waitForProfileRoute` em `auth.setup.ts` cobrem fluxo Google + timeout longo.
- `persistSessionWithPassword` implementa o caminho não-Google; deve ser despriorizado no contrato documental, não necessariamente removido de imediato.

### Established Patterns

- Playwright: projeto `setup` grava `tests/.auth/user.json`; `chromium` depende de `setup` e usa `storageState`.
- `E2E_AUTH_MODE=google` ativa Chrome channel + headed no `playwright.config.ts` para o projeto setup.

### Integration Points

- Callback OAuth Supabase + rota `/login` + botão “Entrar com Google” + destino `/profile` são o caminho crítico para D-01–D-04.
