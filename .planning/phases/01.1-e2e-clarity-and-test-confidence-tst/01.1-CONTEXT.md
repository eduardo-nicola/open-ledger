<domain>
## Phase Boundary

Esta fase **não** acrescenta funcionalidade de produto. Entrega **clareza operacional** dos testes E2E: documentação do fluxo padrão (seed + sessão determinística), separação explícita do modo Google opcional, higiene de tags/grep alinhada aos requisitos AUTH-*, mensagens de erro do setup mais acionáveis e definição do que conta como “pronto” para E2E em PR/CI — dentro do escopo já descrito no roadmap da 01.1 e nos TST-01…TST-07.

**Fora de escopo:** novos cenários E2E de Fases 2+; mudança de ferramenta de teste; performance/load/visual regression.

</domain>

<decisions>
## Implementation Decisions

### Contrato de execução (local + CI)

- **D-01:** O caminho **oficial** (documentado e usado em CI) é **`E2E_AUTH_MODE` ausente ou `password`**, com utilizador de teste criado via **seed SQL** e credenciais fixas já usadas em `auth.setup.ts` (`test@open-ledger.local` + password de teste). Nenhum passo de CI deve depender de OAuth Google real.
- **D-02:** O caminho **`E2E_AUTH_MODE=google`** permanece **opcional**, orientado a desenvolvimento local ou verificação manual; a documentação deve listar envs (`E2E_GOOGLE_TEST_EMAIL`, `E2E_GOOGLE_TEST_PASSWORD`, timeouts) e advertir que **pode ser headed** e frágil em CI — alinhado a TST-02 e TST-03.
- **D-03:** A “matriz modo × comando × quando usar” (TST-01) vive em **`tests/e2e/README.md`** (ou seção equivalente no README raiz se preferirem um único índice — **preferência: arquivo dedicado em `tests/e2e/`** para viver junto do código).

### Scripts npm e semântica de grep

- **D-04:** **`npm run test:e2e:smoke`** é o **gate principal** de fumaça (alinhado ao `@smoke` nos specs autenticados + setup). Documentar isso explicitamente (TST-05).
- **D-05:** **`npm run test:e2e:auth`** documenta-se como execução do **subconjunto** filtrado por `@auth` (grep atual); o README deve explicar a diferença face ao smoke (TST-05). Não é obrigatório alterar o grep nesta fase **desde que** a semântica esteja escrita; mudanças de grep ficam ao critério do plano se reduzirem ruído.
- **D-06:** **`npm run test:e2e:google`** mantém-se como atalho opcional para o modo Google; README indica **não usar** como padrão de CI.

### Tags e rastreabilidade AUTH-*

- **D-07:** **Não reutilizar** o mesmo `@auth-XX` em títulos de testes com asserções diferentes. Opção escolhida: **identificadores únicos por cenário** (ex.: `@auth-04-profile-ui`, `@auth-04-logout-flow`) **e** uma **tabela no README** que mapeia tag → critério de sucesso / requisito REQUIREMENTS — satisfaz TST-04 sem exigir renumerar requisitos de produto.
- **D-08:** Manter `@smoke` nos casos incluídos no gate de fumaça; a tabela de mapeamento deve deixar claro quais tags `@auth-*` estão também em `@smoke`.

### Divisão de trabalho com o plano 01-06 (Fase 1)

- **D-09:** A **01.1** foca em **contrato documental**, **higiene de naming/tags**, **mensagens do setup** e **checklist PR/CI** (TST-01…TST-07). O **`01-06-PLAN.md`** (Wave 6 da Fase 1) continua o sítio natural para **fechar implementação/execução** de cenários E2E pendentes e **checkpoint Google OAuth** quando aplicável. O plano da 01.1 deve **referenciar** o 01-06 para evitar duplicação e, se necessário, **mover** tarefas puramente “escrever assert” para lá.
- **D-10:** Não introduzir novos **projetos** Playwright nesta fase sem justificação no plano; preservar `setup` → `chromium` → `chromium-unauth` salvo análise que prove insuficiência.

### Mensagens e DX (setup / seed)

- **D-11:** Erros em `auth.setup.ts` e caminhos relacionados devem, quando possível, apontar **uma** ação corretiva (“rode `supabase db reset`”, “defina `NEXT_PUBLIC_SUPABASE_URL`”, etc.) — TST-06. Texto orientado a desenvolvedores **em português**, consistente com o resto do repositório.

### Portão de PR / CI

- **D-12:** O README (ou seção de contribuição) deve incluir **checklist mínima**: comando smoke usado no CI, variáveis obrigatórias, e política “Google só fora do gate” — TST-07. O detalhe do **arquivo de CI** exato (GitHub Actions vs outro) fica para o planner/implementação, desde que o critério de “pronto” fique explícito.

### Claude's Discretion

- Nomes exatos das sub-tags (`@auth-04-*`) desde que sejam únicos e a tabela esteja atualizada.
- Organização interna do README (seções, ordem).
- Pequenos ajustes de copy em mensagens de erro desde que mantenham o critério “uma acção corretiva principal”.

</decisions>

## Existing Code Insights

### Reusable Assets

- `tests/e2e/helpers/env.ts` — `loadEnvFromDotEnvLocal()` já usado no setup; manter padrão para qualquer novo helper de env.
- Configuração Playwright existente com `storageState` em `tests/.auth/user.json` após projeto `setup`.

### Established Patterns

- Dois modos em `auth.setup.ts`: API `signInWithPassword` + injecção de cookies via `createServerClient` vs fluxo UI Google com timeouts longos.
- Testes autenticados dependem do projeto `setup`; `chromium-unauth` corre specs sem `storageState`.

### Integration Points

- `npm run dev` como `webServer` do Playwright — documentação deve mencionar custo de arranque e `reuseExistingServer` em local vs CI.
- Seed Supabase deve estar alinhado com credenciais do setup (mensagens de erro devem referir `supabase db reset` quando aplicável).

</code_context>
